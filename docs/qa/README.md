# QA Reports

Storage for QA review reports generated by the qa-reviewer subagent.

## Purpose

This directory contains all QA review reports generated during the development workflow. Reports are created:

1. **Automatically** by `/build` command after TDD completes
2. **Manually** by `/qa` command for targeted reviews

## Naming Convention

`[scope]-YYYYMMDD.md`

### Examples

| Filename | Description |
|----------|-------------|
| `FEAT-018-auth-20260201.md` | Feature-scoped review for FEAT-018 auth work |
| `src-utils-refactor-20260201.md` | Directory-targeted review of src/utils/ |
| `sweep-20260201.md` | Full codebase sweep review |
| `FEAT-018-security-20260201.md` | Second review of same feature (security focus) |

### Scope Types

- **Feature:** `FEAT-XXX-[description]` - Reviews tied to a specific feature
- **Directory:** `[path-slug]` - Reviews of specific directories (slashes → dashes)
- **File:** `[filename-slug]` - Reviews of single files
- **Sweep:** `sweep` - Full codebase review

## Report Status

Each report has a status in its frontmatter:

| Status | Meaning | Action |
|--------|---------|--------|
| `APPROVED` | All checks passed | Ready for `/commit` |
| `NEEDS_FIXES` | Issues found, fixable | Fix issues, re-run `/qa` |
| `BLOCKED` | Critical issues found | Requires human decision |

## Report Lifecycle

```
1. Created by /build (automatic) or /qa (manual)
           ↓
2. Developer reviews findings
           ↓
3. If NEEDS_FIXES:
   - Fix the issues
   - Re-run /qa (creates new report)
   - Loop until APPROVED
           ↓
4. If BLOCKED:
   - Human reviews Tier 2 issues
   - Makes decision in report
   - Re-run /qa after resolution
           ↓
5. APPROVED → /commit unblocked
           ↓
6. After feature ships:
   - Keep for audit trail, or
   - Archive/delete at discretion
```

## Report Template

Reports follow `docs/templates/qa-report-template.md` with sections:

1. **Summary** - Status, issue count, recommendation
2. **Context** - What was reviewed, context source
3. **Security Issues** - OWASP-based findings with confidence
4. **Quality Issues** - Style, complexity, documentation
5. **TDD Validation** - Test results and coverage
6. **Recommended Fixes** - Priority-ordered action items
7. **Approval Gate** - Checklist for approval

## Integration with Workflow

### /build Integration

```
/build FEAT-XXX
    ↓
TDD: RED → GREEN → REFACTOR
    ↓
Handover generated
    ↓
QA reviewer spawns (clean context)
    ↓
docs/qa/FEAT-XXX-[scope]-YYYYMMDD.md created
    ↓
If APPROVED → ready for /commit
If NEEDS_FIXES → fix and re-QA
If BLOCKED → human decision required
```

### /qa Integration

```
/qa [target]
    ↓
Context gathered (handover if feature, git diff otherwise)
    ↓
QA reviewer spawns
    ↓
docs/qa/[target-slug]-YYYYMMDD.md created
```

### /commit Integration

```
/commit
    ↓
Check for recent qa-report matching in-progress feature
    ↓
If BLOCKED or NEEDS_FIXES → warn user
If APPROVED → proceed with commit
```

## Multiple Reports Per Feature

Multiple reports for the same feature are expected and normal:

- `FEAT-018-auth-20260201.md` - Initial review
- `FEAT-018-auth-fix1-20260201.md` - After first round of fixes
- `FEAT-018-security-20260202.md` - Focused security re-review

The most recent APPROVED report is what matters for `/commit`.

## Archival

Reports can be:
- **Kept** for audit trail and historical reference
- **Archived** to `docs/qa/archive/` after feature ships
- **Deleted** if no longer needed

Recommendation: Keep at least the final APPROVED report for each feature.
